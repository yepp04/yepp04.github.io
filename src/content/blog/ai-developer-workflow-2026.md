## 같은 사람, 다른 하루

2020년의 나와 2026년의 나는 같은 개발자입니다. 쓰는 언어도 비슷하고, 풀어야 할 문제의 본질도 크게 달라지지 않았습니다. 그런데 하루 일과가 완전히 달라졌습니다.

| 단계 | 2020년 | 2026년 |
|------|--------|--------|
| 업무 파악 | 기획서 읽고 Jira 티켓 하나하나 수동 생성 | AI에게 기획서를 넘기면 작업 항목을 자동 분해 |
| 기능 개발 | 백엔드 다 짜고 → 프런트 붙이고 → 테스트 작성 | 터미널 여러 개 열어서 각 에이전트가 동시에 진행 |
| 코드 리뷰 | PR 올리고 팀원 리뷰 대기 (반나절~수일) | AI가 PR 올리자마자 리뷰, 사람은 설계 판단에 집중 |
| 버그 수정 | 로그 열어서 눈으로 추적, 재현 환경 구성 | AI가 에러 로그 + 소스를 동시에 읽고 수정안 제안 |

모든 상황이 이렇게 극적으로 달라지진 않습니다. 하지만 **방향은 분명합니다**. 키보드를 두드리는 시간은 줄고, "뭘 만들지"를 정의하고 "제대로 만들어졌는지"를 확인하는 시간이 늘었습니다.

이 글에서는 이 변화를 만든 구체적인 방법들을 정리합니다.

## 바이브코딩의 함정

AI 코딩 도구가 보편화되면서 **바이브코딩(Vibe Coding)** 이라는 패턴이 생겼습니다. 명확한 설계 없이 "그냥 이거 만들어 줘"라고 즉흥적으로 시키는 방식입니다. 결과물이 금방 나오니까 매력적으로 느껴집니다.

실제로 겪어보면 문제가 보입니다. 바이브코딩으로 만든 코드는 일주일 후에 본인이 봐도 이해가 안 됩니다. AI가 짠 코드의 구조를 개발자 본인도 모르기 때문입니다. 버그가 생기면 어디를 고쳐야 하는지 감을 잡을 수 없고, 기능을 추가하려면 처음부터 다시 시키는 게 더 빠릅니다. 켄트 백(Kent Beck)의 표현을 빌리면, 이건 **내년에 심을 씨앗까지 올해 다 먹어버리는 행위**입니다.

프로토타입이나 해커톤에서는 바이브코딩이 빛을 발합니다. 하지만 팀이 유지보수할 프로덕션 코드라면, 개발자의 역할은 코드를 짜는 것이 아니라 **코드의 구조와 품질을 관리하는 것**입니다.

### 그러면 어떻게 해야 하는가

대안은 **Augmented Coding** — 개발자가 주도권을 쥐고 AI를 보조 도구로 쓰는 접근입니다. 켄트 백이 강조하는 핵심은 TDD의 Red-Green-Refactor 사이클을 AI와 함께 돌리는 것입니다.

1. **Red** — 실패하는 테스트를 먼저 작성합니다. "무엇이 동작해야 하는가"를 코드로 정의하는 단계입니다.
2. **Green** — 테스트를 통과할 최소한의 코드만 AI에게 시킵니다.
3. **Refactor** — 동작은 유지하면서 구조를 개선합니다.

테스트가 안전망이 됩니다. AI가 어떤 코드를 짜든, 테스트를 통과하면 의도대로 동작한다는 보장이 있습니다. 리팩토링할 때도 테스트가 깨지면 바로 알 수 있습니다.

```
# 이렇게 하지 마세요
"주문 취소 기능 만들어줘"

# 이렇게 하세요
"주문 상태가 '배송중'이면 취소가 불가능한 테스트를 작성해줘"
→ 테스트 확인
"이 테스트를 통과하는 cancelOrder 함수를 구현해줘"
→ 통과 확인
"부분 취소, 환불 금액 계산 테스트를 추가해줘"
→ 반복
```

이 사이클이 느리게 느껴질 수 있지만, 3개월 후에 "이 코드 왜 이러지?"라며 처음부터 다시 짜는 것보다 확실히 빠릅니다.

## Plan First: 삽질의 80%는 계획 부재에서 온다

AI에게 바로 코드를 시키면 높은 확률로 삽질이 시작됩니다. AI는 "시키면 바로 하는" 특성이 있어서, 방향이 잘못되어도 멈추지 않고 코드를 쏟아냅니다. 한참 진행된 후에야 방향이 틀렸다는 걸 깨닫게 되면 되돌리기가 어렵습니다.

### SDD: 설계 문서부터 쓰기

**SDD(Spec Driven Development)** 는 코드 작성 전에 '무엇을, 어떻게' 만들지 **설계 문서를 먼저 작성**하고 AI가 이를 따르게 하는 접근법입니다.

```
# 1단계: 설계
"상품 재고 관리 기능을 리뉴얼하려고 해.
현재는 단일 창고 기준인데, 다중 창고를 지원해야 해.
창고 간 재고 이동과 실시간 재고 조회가 핵심이야.
구현 전에 설계 문서부터 작성해줘.
데이터 모델, API 목록, 상태 전이 다이어그램 포함해서."

# 2단계: 검토
→ 설계 문서를 직접 읽고 검토. "재고 이동 중 동시성 문제는 어떻게 처리하지?"
  같은 빠진 부분을 잡아서 수정 지시.

# 3단계: 구현
"확정된 설계 문서대로 구현해줘. 설계에 없는 기능은 추가하지 마."
```

설계 문서라고 해서 거창한 게 아닙니다. "어떤 API가 필요하고, 데이터가 어떻게 흐르는지"를 텍스트로 정리하는 것만으로 충분합니다. 이게 있으면 AI가 엉뚱한 방향으로 달려가는 것을 막을 수 있고, 나중에 코드를 리뷰할 때도 "설계대로 구현되었는가"라는 명확한 기준이 생깁니다.

### Plan Mode: AI에게 질문을 먼저 시키기

Claude Code에는 `Shift+Tab`으로 진입하는 **플랜 모드**가 있습니다. 이 모드에서 AI는 코드를 짜지 않고, 대신 질문을 합니다.

```
[Plan Mode]
> 검색 기능을 개선하고 싶어

Claude: 몇 가지 확인하겠습니다.
1. 현재 검색은 DB 풀텍스트인가요, 별도 검색 엔진(Elasticsearch 등)인가요?
2. 자동완성이나 오타 보정도 범위에 포함인가요?
3. 검색 대상이 상품만인가요, 게시글이나 사용자도 포함인가요?
4. 검색 결과 정렬 기준을 커스터마이징할 수 있어야 하나요?
```

질문에 답하다 보면 자신도 모르게 **요구사항이 명확해집니다**. "검색 개선"이라는 모호한 요청이 "Elasticsearch 도입, 상품+게시글 통합 검색, 자동완성 지원, 관련도순 정렬"이라는 구체적 명세가 됩니다.

이 과정이 끝나면 AI가 확정된 계획을 바탕으로 개발을 진행합니다. 삽질이 극적으로 줄어듭니다.

## MCP: AI에게 눈과 손을 달아주기

AI 코딩 도구의 가장 큰 변화 중 하나는 **MCP(Model Context Protocol)** 입니다. AI가 코드만 읽는 게 아니라, 외부 서비스의 데이터를 직접 보고 조작할 수 있게 됩니다.

모든 MCP를 다 쓸 필요는 없습니다. 실무에서 체감이 큰 것들만 추리면 이렇습니다.

### 장애 대응: Sentry MCP

금요일 오후, 슬랙에 알림이 옵니다. 주문 완료 API에서 간헐적 500 에러. 예전이라면 Sentry 대시보드를 열고, 스택 트레이스를 복사하고, 관련 코드를 하나하나 찾아가며 원인을 좁혀갔을 겁니다.

지금은 이렇게 합니다.

```
"Sentry에서 OrderController 관련 최근 에러를 확인하고,
스택 트레이스에 나온 코드 경로를 직접 읽어서 원인을 분석해줘"
```

AI가 에러 로그와 소스코드를 동시에 읽으면서 "이 분기에서 null 체크가 빠져있다"는 수준의 원인 분석과 수정안을 제시합니다. 물론 여러 서비스가 얽힌 환경에서는 AI의 수정안이 사이드이펙트 없이 안전한지 직접 확인해야 합니다. 하지만 "어디를 봐야 하는지"를 찾는 시간이 확실히 줄어듭니다.

### 업무 자동화: Atlassian MCP

스프린트 플래닝 전날, PM이 기획서를 공유합니다. 이걸 Jira 티켓으로 옮기는 작업 — 에픽 생성, 스토리 분해, 서브태스크 쪼개기, 스토리 포인트 산정 — 이 작업 자체가 하나의 업무였습니다.

```
"이 기획서를 분석해서 Jira 티켓으로 분해해줘.
백엔드/프런트/인프라 라벨을 달고, 의존성 관계도 설정해."
```

직접 하면 30분~1시간, AI에게 시키면 초안이 몇 분 안에 나옵니다. 물론 AI가 산정한 스토리 포인트나 우선순위는 조정이 필요하지만, 빈 화면에서 시작하는 것과 초안을 다듬는 것은 체감이 다릅니다.

### 환각 방지: Context7

AI에게 코드를 시키다 보면 한 번쯤 겪는 상황이 있습니다. AI가 자신 있게 작성한 코드가 빌드에서 에러를 뱉는 겁니다. 확인해보면 6개월 전에 deprecated된 API를 쓰고 있거나, 아예 존재하지 않는 메서드를 호출하고 있습니다. AI는 학습 시점의 데이터를 기반으로 코드를 작성하기 때문에, 빠르게 변하는 프레임워크에서 이런 **환각(Hallucination)** 이 발생합니다.

**Context7** 같은 문서 참조 MCP를 연결하면, AI가 코드를 작성할 때 **해당 라이브러리의 최신 공식 문서를 실시간으로 확인**합니다. "이 훅의 시그니처가 바뀌었나?" 같은 검증을 AI가 스스로 하게 되는 셈입니다.

### UI 검증: Playwright / Browser MCP

AI가 프런트엔드 코드를 짜면 "코드상으로는 맞는 것 같은데 실제로 어떻게 보이지?"라는 의문이 남습니다. Playwright나 Browser MCP를 연결하면 AI가 **실제 브라우저에서 결과물을 렌더링하고 스크린샷으로 확인**합니다.

이것을 **Self-verification** 이라고 합니다. AI가 자기가 짠 코드의 결과를 직접 보고, 버튼이 잘려 있거나 레이아웃이 깨져 있으면 스스로 수정합니다. "눈이 없는 AI"에게 눈을 달아주는 것만으로 결과물의 완성도가 확연히 달라집니다.

### MCP 관리 원칙

MCP는 많다고 좋은 게 아닙니다. 도구가 너무 많으면 AI가 "이 상황에서 뭘 써야 하지?"를 고르는 데 토큰을 쓰고, 정작 작업 품질은 떨어집니다. 연장통에 렌치가 200개 들어있으면 오히려 작업이 느려지는 것과 같습니다.

상시로 켜두는 MCP는 자신의 워크플로우에 맞는 핵심 몇 개만 유지하고, 나머지는 필요한 프로젝트에서만 활성화하는 것이 좋습니다. `gh pr create` 한 줄이면 되는 작업에 GitHub MCP를 거칠 이유도 없습니다.

## 컨텍스트 관리: 숨은 생산성 킬러

AI 코딩 도구를 쓰다 보면 체감하게 되는 문제가 있습니다. 세션이 길어질수록 AI의 응답 품질이 떨어집니다. 이전 대화 내용과 새로운 요청이 섞이면서 컨텍스트가 오염되기 때문입니다. AI는 "지금 뭘 하고 있는지" 혼란스러워하고, 엉뚱한 파일을 수정하거나 이미 해결한 문제를 다시 해결하려고 합니다.

### 기능 단위로 세션을 끊어라

가장 효과적인 방법은 **기능 하나가 끝나면 세션을 초기화**하는 것입니다.

```
# 기능 A 개발 완료
/clear

# 기능 B 시작 (깨끗한 상태에서)
"사용자 프로필 편집 기능을 추가해줘..."
```

`/clear`로 대화를 완전히 초기화하면 컨텍스트 오염 없이 새로 시작합니다. `/compact`는 대화를 요약해서 핵심만 남기는 옵션인데, 완전 초기화보다는 덜 효과적이지만 이어서 작업할 때 유용합니다.

### CLAUDE.md: 세션을 넘어 살아남는 맥락

세션을 초기화하면 이전 맥락이 다 날아갑니다. 그래서 **프로젝트의 핵심 정보**는 `CLAUDE.md`에 적어두는 것이 중요합니다. Claude Code는 매 세션 시작 시 이 파일을 가장 먼저 읽습니다.

```markdown
# CLAUDE.md
## 프로젝트 개요
사내 주문 관리 시스템. Spring Boot + React + PostgreSQL.

## 아키텍처
- 백엔드: src/main/ 아래 controller → service → repository 계층
- 프런트: src/webapp/ 아래 pages → components → hooks 구조

## 컨벤션
- REST API는 /api/v2/ 접두사
- 커밋 메시지는 Conventional Commits
- 테스트는 given-when-then 패턴

## 주의사항
- OrderService.calculateDiscount()는 절대 수정하지 말 것 (레거시 연동)
- prod 환경에서는 batch job이 02시에 동작하므로 관련 테이블 스키마 변경 시 주의
```

이 파일의 핵심은 **AI가 실수할 때마다 업데이트하는 것**입니다. AI가 `/api/v1/`으로 엔드포인트를 만들었다면, CLAUDE.md에 "REST API는 /api/v2/ 접두사"를 추가합니다. 이러면 같은 실수가 반복되지 않습니다. 이것이 **복리 학습**입니다 — 매 실수가 영구적인 개선으로 축적됩니다.

### GitHub Issue를 맥락 저장소로 활용하기

복잡한 기능이나 장기 프로젝트는 한 세션에서 끝나지 않습니다. 이때 **GitHub Issue에 계획과 진행 상황을 기록**해두면, 새 세션에서 AI가 이를 읽고 이어서 작업할 수 있습니다.

```
"GitHub Issue #47을 읽고 이어서 작업해줘.
3단계 '알림 발송 로직 구현'부터 시작해."
```

## AI 코드 리뷰: PR이 올라간 순간 리뷰가 시작된다

예전에는 PR을 올리면 팀원이 시간 날 때까지 기다려야 했습니다. 급한 핫픽스인데 리뷰어가 회의 중이면 반나절이 날아갑니다. AI 리뷰어를 붙여두면 이 병목이 사라집니다.

### 세팅: 두 AI 리뷰어 붙이기

실무에서 효과적인 패턴은 **성격이 다른 AI 리뷰어 두 개를 동시에 붙이는 것**입니다. 세팅은 생각보다 간단합니다.

**1단계: CodeRabbit 연결**

[CodeRabbit](https://coderabbit.ai)은 GitHub App으로 설치합니다. GitHub Marketplace에서 "CodeRabbit"을 검색하고 설치하면, 이후 PR이 올라올 때마다 자동으로 리뷰 코멘트를 남깁니다. 별도 설정 없이 바로 동작합니다. 오픈소스 프로젝트는 무료입니다.

변경된 파일 단위로 라인별 코멘트를 남기고, PR 전체의 요약("이 PR은 인증 로직을 리팩토링하고 세션 타임아웃 처리를 추가합니다")도 자동 작성합니다.

**2단계: Claude를 PR 리뷰어로 추가**

Claude Code에서 PR 리뷰를 돌리는 방법은 두 가지입니다.

로컬에서 직접 리뷰를 요청하는 방식:

```bash
# 현재 브랜치의 변경사항을 리뷰
claude "이 브랜치의 변경사항을 리뷰해줘. 보안, 성능, 에러 핸들링 관점에서."
```

또는 GitHub Actions에 Claude Code를 연결해서, PR이 올라올 때마다 자동으로 리뷰가 달리게 할 수도 있습니다.

```yaml
# .github/workflows/ai-review.yml
name: AI Code Review
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@beta
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          direct_prompt: |
            이 PR의 변경사항을 리뷰해줘.
            보안 취약점, 성능 문제, 에러 핸들링 누락을 중점적으로 확인해.
```

이렇게 하면 PR을 올리는 순간 CodeRabbit과 Claude가 **동시에 각자의 리뷰를 시작**합니다.

### 왜 두 개를 쓰는가

둘이 잡아내는 영역이 다릅니다.

| 관점 | 잘 잡는 것 | 예시 |
|------|-----------|------|
| **패턴/구조** | 코드 스멜, 중복, 네이밍 | "이 로직이 세 곳에 반복됩니다. 헬퍼로 추출하세요." |
| **런타임 위험** | N+1 쿼리, 레이스 컨디션, 메모리 누수 | "반복문 안에서 DB 호출이 발생합니다." |
| **일관성** | API 응답 형식, 에러 코드, 컨벤션 위반 | "다른 엔드포인트는 404를 반환하는데 여기만 400입니다." |
| **보안** | 인증 누락, 입력 검증, 민감 정보 노출 | "사용자 입력이 검증 없이 쿼리에 들어갑니다." |

한 명의 리뷰어가 모든 관점을 커버하긴 어렵습니다. 두 AI가 각자 리뷰를 남기면, 하나가 놓친 것을 다른 하나가 잡아낼 확률이 높아집니다.

### 크로스 체크: AI 리뷰를 다시 AI로 걸러내기

AI 리뷰어의 문제는 **노이즈가 많다**는 것입니다. 진짜 버그 지적 사이에 "이 변수명을 더 서술적으로 바꾸세요" 같은 취향 코멘트가 섞여 있습니다. 코멘트가 20개인데 그중 중요한 게 3개라면, 나머지 17개를 읽는 시간이 아깝습니다.

여기서 한 단계를 더 추가합니다. PR에 두 AI의 리뷰가 달리면, **로컬에서 Claude Code를 열고** 해당 PR의 코멘트를 읽게 시킵니다.

```bash
# Claude Code에서 GitHub CLI를 통해 PR 코멘트를 읽고 분류
claude "gh pr view 142 --comments 결과를 읽고,
두 리뷰어의 코멘트를 종합해서 분류해줘.
1. 반드시 수정해야 하는 것 (버그, 보안, 성능)
2. 수정하면 좋은 것 (가독성, 일관성)
3. 취향 차이 (무시해도 되는 것)"
```

Claude Code가 GitHub CLI로 PR의 코멘트를 가져와서, 전체 리뷰를 우선순위별로 정리해줍니다. 개발자는 1번 카테고리만 집중적으로 보면 됩니다. 코멘트 20개를 하나하나 읽는 대신, **중요도가 매겨진 요약**을 받는 셈입니다.

### 인간은 무엇을 봐야 하는가

AI가 잘 잡는 것과 못 잡는 것이 명확히 나뉩니다.

**AI가 잘하는 것:**
- 문법적 오류, 타입 불일치, 미사용 변수
- 알려진 패턴의 성능 문제 (N+1, 불필요한 재렌더링)
- 프로젝트 컨벤션 위반 (네이밍, 포맷, import 순서)

**인간이 해야 하는 것:**
- "이 기능이 요구사항을 충족하는가?"
- "이 설계가 3개월 후 확장 계획과 충돌하지 않는가?"
- "이 변경이 다른 팀이 쓰는 API에 영향을 주는가?"
- "이 비즈니스 로직이 정책적으로 맞는가?"

AI 리뷰가 도입되면 인간 리뷰어의 역할이 바뀝니다. 세미콜론을 찾는 사람에서, **설계와 비즈니스 로직을 판단하는 사람**으로. 이게 원래 코드 리뷰가 해야 할 일이었는데, 그동안 사소한 것들에 시간을 빼앗기고 있었던 겁니다.

## 반드시 피해야 할 안티패턴

### AI 코드 맹신

AI가 만든 코드를 diff도 안 보고 커밋하는 건 위험합니다. 흔히 보고되는 사례로, AI가 인증 미들웨어를 리팩토링하면서 특정 엔드포인트의 권한 체크를 빼먹는 경우가 있습니다. 코드 자체는 깔끔한데, 비로그인 사용자가 관리자 API를 호출할 수 있는 구멍이 생기는 식입니다. AI는 "돌아가는 코드"는 잘 만들지만, 그 코드가 **의도하지 않은 것을 허용하는지**는 판단하지 못합니다.

**원칙: AI를 믿되, 검증하라.**

### 긴 세션에서의 컨텍스트 오염

한 세션에서 너무 오래 대화하면 AI가 이전 내용과 현재 요청을 혼동합니다. 로그인 기능을 작업하다가 장바구니 기능으로 넘어갔는데, AI가 장바구니 코드에 로그인 관련 로직을 섞어 넣는 식입니다. 건드리지 말라고 한 파일을 슬쩍 수정하는 것도 이 증상의 일종입니다.

**원칙: 기능 하나가 끝나면 `/clear`.**

### 환각에 의한 유령 API

앞서 MCP 섹션에서도 다뤘지만 한 번 더 강조할 만합니다. React Server Components의 `"use server"` 디렉티브를 잘못된 위치에 넣거나, Prisma 5에서 변경된 트랜잭션 API를 Prisma 4 문법으로 쓰는 경우가 실제로 발생합니다.

**원칙: 최신 프레임워크 작업 시 Context7 같은 문서 주입 도구를 반드시 사용하라.**

### 과도한 도구 설치

MCP를 잔뜩 깔아놓으면 AI가 도구를 고르는 데 토큰을 쓰느라 정작 코드 품질이 떨어집니다. 실제로 MCP를 20개 넘게 활성화했더니 간단한 파일 수정에도 불필요한 도구를 끌어다 쓰려는 행동이 관찰된 적이 있습니다.

**원칙: 자신의 핵심 워크플로우에 필요한 것만 상시 활성화. 나머지는 프로젝트별로 필요할 때만.**

## 개발자의 역할이 바뀌고 있다

2020년의 개발자는 코드를 직접 짜는 사람이었습니다. 2026년의 개발자는 **AI 에이전트 팀을 지휘하는 오케스트레이터**에 가깝습니다.

달라진 것:
- 코드를 직접 타이핑하는 시간이 줄었습니다.
- 설계를 명확히 하고, 적절한 지시를 내리고, 결과를 검증하는 시간이 늘었습니다.
- 여러 AI 에이전트를 병렬로 운용하는 능력이 생산성을 좌우합니다.

달라지지 않은 것:
- 좋은 코드와 나쁜 코드를 구분하는 안목은 여전히 개발자의 몫입니다.
- 비즈니스 요구사항을 기술적 해법으로 번역하는 능력은 여전히 핵심입니다.
- 수십 개 서비스가 얽힌 환경에서 수정의 영향 범위를 판단하는 것은 AI가 대신할 수 없습니다.

도구는 바뀌었지만, 개발자의 본질적인 가치 — **문제를 정의하고, 해법을 설계하고, 품질을 보증하는 것** — 는 그대로입니다. 오히려 AI가 단순 작업을 대신하게 되면서, 이 본질적인 능력의 중요성이 더 커졌습니다.
